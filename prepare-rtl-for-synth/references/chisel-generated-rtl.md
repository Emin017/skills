# Chisel Generated RTL

Use this reference when the RTL appears to be generated by Chisel, FIRRTL, or CIRCT `firtool`. Generated comments, `_T` temporaries, `firtool` version banners, `.fir` sources, `.anno.json` files, and `*.conf` memory files are common indicators.

## Syntax Compatibility Changes

Chisel-generated Verilog/SystemVerilog syntax can vary with `firtool` options. If yosys-slang, Yosys, or another EDA parser fails on otherwise correct generated RTL, check whether the project can regenerate RTL with more compatible `firtool` lowering options instead of patching generated files by hand.

Do not guess Chisel compiler or `firtool` option names from memory. Always confirm the installed tool's supported options before changing build commands:

```bash
firtool --help
firtool --help-hidden
```

If the project invokes Chisel through `sbt`, mill, Bazel, or a custom script, inspect that wrapper and run its documented help command when available. Option names and defaults can differ across Chisel/CIRCT versions, so only use options confirmed in the local flow.

Refer to CIRCT's official Verilog generation guide:

https://circt.llvm.org/docs/VerilogGeneration/

The CIRCT guide describes `--lowering-options=` and tool-targeted compatibility options. For Yosys-oriented flows, it specifically recommends options such as `disallowLocalVariables` and `disallowPackedArrays`.

Record any code-generation option changes in the manifest or task summary, because regenerated Chisel RTL can overwrite manual edits.

## Macro Replace Preparation

### Do Not Replace Every Behavioral Memory

Do not assume every generated behavioral memory model should become a foundry SRAM macro.

- If the Chisel source uses `Mem`, ROMs, small register-bank memories, or generated behavioral arrays without `SyncReadMem`/`SRAM` intent, the behavioral RTL may be the intended implementation.
- If the Chisel flow did not enable sequential memory replacement, generated memory behavior may be deliberate and should be preserved.
- Confirm intent with the user, source Chisel, generator documentation, or existing build scripts before replacing generated arrays.

Chisel documents that memory abstractions may map either to simple Verilog behavioral descriptions or to external memory-generator modules, and that `SyncReadMem` is the construct usually intended for synchronous SRAM-like memories. Use that distinction when deciding replacement scope.

### Check for `repl-seq-mem` Output

When Chisel/firtool code generation uses a sequential-memory replacement option such as `repl-seq-mem`, the build may emit one or more `.conf` files. Treat these files as first-class replacement inputs.

Before deriving widths and depths from generated Verilog alone, search for:

```bash
rg -n "repl-seq-mem|seq-mem|SyncReadMem|SRAM|firtool|\\.conf" .
rg --files -g "*.conf" -g "*.fir" -g "*.anno.json" -g "*.sv" -g "*.v"
```

Use `.conf` contents to populate the replacement manifest when present. They often encode the generated memory name, depth, width, mask granularity, and port shape more directly than the emitted RTL.
